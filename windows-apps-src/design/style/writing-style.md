# <a name="writing-style"></a>撰寫方式

您表達錯誤訊息的方式、撰寫說明文件的方式，甚至是您為按鈕選擇的文字，對於應用程式的可用性都有很大的影響。 

撰寫方式可以在糟糕的使用者體驗...

![Windows 7 當機藍色畫面](images/writing/bluescreen_old.png)

...和良好的體驗之間產生巨大差異。

![Windows 10 當機藍色畫面](images/writing/bluescreen_new.png)


## <a name="writing-with-a-natural-voice-and-tone"></a>使用自然的語氣和語調撰寫

研究顯示，使用者對於友善、有幫助且精簡的書寫風格反應最好。 

## <a name="be-friendly"></a>友善

最重要的是，您不會想嚇跑使用者。 態度放輕鬆、不要太官腔，也不要使用使用者不了解的術語。 即使運作中斷，也不要怪罪使用者。 您的應用程式應該負起責任，並提供將使用者行動置於優先考量的熱情指導。

![之前：發生錯誤，無法上傳您的影像。 如果後續嘗試同樣失敗，應用程式可能需要重新初始化。 之後：因為發生錯誤，我們無法上傳影像。 請再試一次 - 如果仍然無法運作，請嘗試重新啟動應用程式。](images/writing/friendly_example.png)

## <a name="be-helpful"></a>有幫助

永遠將重點放在解釋發生了什麼事，並提供使用者需要的相關資訊，而不是用沒必要的資訊來淹沒使用者。

![之前：無法連線到網路。 之後：您的裝置已連線到網路，但網路未連線到網際網路。 請嘗試將您的路由器或數據機重新開機。 如果您使用公用熱點，請確定您已登入。](images/writing/helpful_example.png)

## <a name="be-clear-and-concise"></a>清楚、簡潔

大部分時候，文字不是應用程式的重點。 文字是用來引導使用者，教導他們發生了什麼事以及接下來應該做什麼。 撰寫應用程式中的文字時，請勿忽視這一點。 使用您的對象熟悉的語言。 通常這表示選擇日常口語文字，但特殊用途的應用程式會有自己的標準。 不應讓使用者猜測您的文字代表的意義，所以如果您不確定該使用什麼語調，請保持簡單。

![之前：發生問題，因此我們無法儲存您的檔案。 此裝置可能無法存取該位置，或可能空間不足。 之後：我們無法將您的檔案儲存到該位置。 請使用不同的位置，再試一次。](images/writing/concise_example.png)

## <a name="writing-tips"></a>撰寫技巧

在應用程式的有限空間中，有很多方法可以實現這些一般原則。 以下是一些我們覺得有助於將這些原則付諸行動的策略：

### <a name="lead-with-whats-important"></a>以重要的資訊起頭

使用者必須在一瞥之下就能閱讀並了解您的文字。 不要用不必要的介紹來裝填文字。 讓關鍵要點最顯而易見，一律最先呈現概念的核心。

**正確：** 選取 \[濾鏡\] 以新增效果至影像。

**錯誤：** 如果您想要新增視覺效果或改變影像，請選取 \[濾鏡\]。

### <a name="emphasize-action"></a>強調動作

應用程式由動作來定義。 使用者在使用應用程式時會採取動作，而應用程式會採取動作來回應使用者。 請確定您的文字在整個應用程式中都使用*主動語態*。 應該將使用者和功能描述為正在執行動作，而不是這些動作由他們完成。

**正確：** 重新啟動應用程式以查看您的變更。

**錯誤：** 當應用程式重新啟動時，將會套用所做的變更。

### <a name="short-and-sweet"></a>短而甜

使用者會掃視文字，通常會整個忽略較大的文字區塊。 不需犧牲必要的資訊和呈現，但不要使用超過所需的文字量。 有時候，這表示傾向採用許多短句子或片段。 其他時候，這表示要對較長句子的文字和結構更加挑剔。

**正確：** 抱歉，我們無法上傳圖片。 如果再度發生這種情況，請嘗試重新啟動應用程式。 但請不要擔心 — 當您回來時，您的圖片已經準備好了。

**錯誤：** 發生錯誤，我們無法上傳圖片。 請再試一次，如果再次遇到這個問題，您可能需要重新啟動應用程式。 但不用擔心 - 我們已經將您的工作儲存在本機上，當您回來時，它已經準備好了。

## <a name="style-conventions"></a>風格慣例

如果您不認為自己是寫手，嘗試實施這些原則和建議可能會讓人卻步。 但請勿擔心 — 使用簡單直白的語言，是提供良好使用者體驗的絕佳方式。 如果您仍然不確定如何組織您的文字，以下是一些很有幫助的指導方針。

### <a name="addressing-the-user"></a>把使用者當成說話對象

直接向使用者講話。

* 一律將使用者稱呼為「您」。

* 使用「我們」代表您自己的觀點。 語氣應該是熱情的，並且可讓使用者感受屬於體驗的一部份。

* 不要使用「我」來指稱應用程式的觀點，即使應用程式的創作者只有您一人。

> 我們無法將您的檔案儲存到該位置。

### <a name="abbreviations"></a>縮寫 (Abbreviations)

當您需要在應用程式中多次提及某個產品、地點或技術概念，縮寫非常有用。 縮寫可以節省空間，感覺更自然，前提是使用者了解該術語。

* 不要假設使用者已經熟悉任何縮寫，就算您認為該縮寫已很常見。

* 使用者第一次看到新縮寫時，一定要定義其代表意義。

* 不要使用與其他縮寫太過類似的縮寫。

> 通用 Windows 平台 (UWP) 設計指導方針可以協助您設計和建立美觀、優雅的應用程式。 運用每個 UWP app 中所包含的設計功能，您可以建置可在一系列裝置上縮放的使用者介面 (UI)。

### <a name="contractions"></a>縮寫 (Contractions)

大家已經習慣縮寫，也覺得看到縮寫很正常。 避免使用縮寫反而會讓您的應用程式看起來太過正式，甚至呆板。

* 請依使用習慣，自然使用縮寫即可。

* 不要為了節省空間而使用不常見的縮寫，或讓您的措辭感覺不自然。

> 當您滿意您的影像，請按 \[儲存\] 將它新增到您的影像中心。 在那裡，您可以將它分享給朋友。

### <a name="periods"></a>句點

文字結尾有句點表示該文字是完整的句子。 一大段文字請使用句點，比一般句子短的文字請避免使用句點。

* 在工具提示、錯誤訊息和對話方塊中，使用句點來結束完整的句子。

* 按鈕、選項按鈕、標籤或核取方塊的文字不要使用句點。

> **您未連線**
> * 請檢查網路纜線是否插上。
> * 請確定您不是處於飛航模式。
> * 查看無線開關是否已開啟。
> * 將路由器重新開機。

### <a name="capitalization"></a>大寫

大寫字母代表重要，因此很容易被濫用。

* 專有名詞應大寫。

* 應用程式中每個文字字串的開頭 - 每個句子、標籤和標題的開頭 - 要大寫。

> **哪個部分發生問題？**
> * 忘記密碼
> * 系統不接受我的密碼
> * 可能有其他人正在使用我的帳戶

## <a name="error-messages"></a>錯誤訊息

當您的應用程式發生問題，使用者會密切注意。 因為當使用者遇到錯誤訊息時會感到混淆或挫折，因此在這個區域，良好的語氣和語調會產生非常大的影響。

更重要的是，您的錯誤訊息不能怪罪使用者。 但也請不要用使用者不了解的資訊來轟炸他們。 大部分時候，遇到錯誤的使用者只想要趕快回到他們本來正在做的事情，越快越輕鬆越好。 因此，您撰寫的錯誤訊息應該：

* **友善**：負起發生問題的責任，避免使用不熟悉的詞彙和技術術語。

* **有幫助**：盡您最大的能力告訴使用者發生哪些錯誤，告訴他們接下來會發生什麼事情，並提供他們可以完成的實際解決方案。

* **清楚、簡潔**：排除無關的資訊。

![良好的錯誤訊息範例](images/writing/connection_error.png)

## <a name="buttons"></a>按鈕

按鈕上的文字必須夠簡潔，讓使用者一眼就能讀完，也必須夠清楚，讓按鈕的功能顯而易見。 按鈕上文字的長度上限應該只是幾組簡短文字，大部分時候應該更短。

撰寫按鈕的文字時，請記住每個按鈕代表一個動作。 按鈕的文字請務必使用*主動語態*，並使用代表動作 (而非反應) 的文字。

![良好按鈕的範例](images/writing/install_button.png)

## <a name="dialogs"></a>對話方塊

撰寫錯誤訊息的許多建議也同樣適用於為您的應用程式建立任何對話方塊的文字。 即使使用者已預期會出現對話方塊，但對話方塊仍會中斷應用程式的一般流程，因此內容必須精簡有幫助，讓使用者能夠返回原本所進行的活動。

但最重要的是對話方塊的標題和其按鈕之間的「呼叫與回應」。 請確定您的按鈕是標題所張貼問題的明確答案，並且應用程式中所有按鈕的格式都一致。

![良好對話方塊的範例](images/writing/password_dialog.png)

## <a name="spoken-experiences"></a>口語體驗

撰寫口語體驗 (例如 Cortana) 的文字時，也適用相同的一般原則和建議。 在這些功能中，良好的寫作原則更顯重要，因為您無法提供其他視覺設計元素給使用者來補充口語文字。

* **友善**：以交談語調和使用者互動。 跟任何其他區域相比，口語體驗更需要聽起來溫暖且平易近人，讓使用者不會害怕與其交談。

* **有幫助**：當使用者詢問不可能的事項，能提供替代建議。 跟錯誤訊息一樣，如果發生問題而您的應用程式無法滿足要求，則應改為提供可讓使用者嘗試詢問的實際替代方案。

* **清楚、簡潔**：使用簡單的語言。 口語體驗不適合冗長的句子或複雜的文字。

## <a name="accessibility-and-localization"></a>協助工具和當地語系化

撰寫應用程式時如果考慮到協助工具和當地語系化，您的應用程式可以適用於更多使用者。 光靠文字無法達成這個目的，不過直接、友善的語言是個好起點。 如需詳細資訊，請參閱我們的[協助工具概觀](https://docs.microsoft.com/windows/uwp/design/accessibility/accessibility-overview)和[當地語系化指導方針](https://docs.microsoft.com/windows/uwp/design/globalizing/globalizing-portal)。

* **友善、有幫助**：將不同的體驗納入考量。 避免使用對國際使用者沒有意義的片語，也不要在文字中假設使用者做得到什麼、做不到什麼。

* **清楚、簡潔**：避免使用不必要的特殊及專業詞彙。 您的文字越直白，就越容易當地語系化。


## <a name="techniques-for-non-writers"></a>非寫手的技巧

不需要是訓練有素或經驗豐富的作家，也能為使用者提供良好的體驗。 選擇您聽起來舒服的文字，那麼使用者也會覺得舒服。 不過，有時候知易行難。 如果您遇到困難，以下技巧可以幫助您。 

* 請想像您正在向朋友談論您的應用程式。 您會如何向他們解釋應用程式？ 您如何討論功能？如何給予指示？ 更好的方法是，向尚未用過應用程式的人解說您的應用程式。 

* 想像您會如何描述完全不同的應用程式。 舉例來說，如果您正在製作遊戲，就想像您會如何口說或書寫來描述財務或新聞應用程式。 您所使用語言和結構的對比，可以讓您更深入地了解實際撰寫項目的正確措辭。

* 查看類似的應用程式以尋找靈感。 

很多人可能會卡在尋找正確的字詞，所以如果對於某些事就是感覺不對而遲遲無法確定，不用覺得難過。
